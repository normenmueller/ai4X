# Solution Realization Architecture (MUST)

## Purpose

Shape a concrete solution change into an interoperable topology and implementable transition architecture without losing architectural intent.

## Trigger

Use when a concrete solution must be specified, staged, and governed from requirements through target roadmap and implementation change.

## Rules (MUST)

- Derive solution structure from explicit requirements and superior architectures.
- Make systems, subsystems, components, and interoperability explicit before implementation detail takes over.
- Stage major change through target roadmaps and implementable transition architectures.
- Provide risk-based governance for deviations during implementation without dissolving architecture accountability.

## Fallback

- If requirements, superior architecture constraints, or delivery boundaries are unclear, keep the output provisional and name the missing realization decisions.

## Minimal Output Contract

- solution scope and requirements
- topology or interoperability view
- transition roadmap
- implementation governance note
