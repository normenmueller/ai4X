# Data Governance Architecture (MUST)

## Purpose

Shape data-domain architecture through explicit governance, integration, migration, protection, and lifecycle control.

## Trigger

Use when a segment or domain architecture task is data-focused and must govern data ownership, flow, protection, integration, or migration across the landscape.

## Rules (MUST)

- Make data ownership, governance, integration, migration, and protection responsibilities explicit.
- Distinguish data architecture from application behavior, reporting convenience, and transport format details.
- Keep data quality, security, and lifecycle consequences explicit for every material architecture change.
- Evaluate data change as architecture of the data landscape, not only as a local schema delta.

## Fallback

- If authoritative data boundaries, governance owners, or integration scope are unclear, keep the output provisional and name the missing data architecture decisions.

## Minimal Output Contract

- data scope and ownership
- governance or protection implications
- integration or migration implications
- open data architecture risks
