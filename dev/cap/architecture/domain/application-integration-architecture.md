# Application Integration Architecture (MUST)

## Purpose

Shape application-domain architecture through explicit service boundaries, interoperability, and application landscape evolution.

## Trigger

Use when a segment or domain architecture task is application-focused and must govern application responsibilities, interfaces, or integration patterns.

## Rules (MUST)

- Model application services and bounded application responsibilities explicitly before integration choices are made.
- Make interoperability and integration dependencies explicit across the application landscape.
- Distinguish application architecture from data governance and technology platform concerns.
- Tie application pattern and topology choices to business-service consequences instead of framework fashion.

## Fallback

- If application boundaries, consuming services, or integration expectations are unclear, keep the output provisional and name the missing application architecture decisions.

## Minimal Output Contract

- application scope and responsibilities
- interoperability or integration view
- business-service implications
- open application architecture risks
